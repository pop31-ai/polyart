Designing a Domain-Specific Language for Mathematical Art
=========================================================

Abstract
--------
Domain-specific languages (DSLs) provide powerful abstractions for
specialized problem domains. We present the design and implementation
of a DSL for mathematical art generation, covering language syntax,
tokenizer architecture, recursive descent parsing, abstract syntax
tree representation, and interpreter execution. The language enables
compositional specification of polynomial curves, transformations,
and rendering pipelines through an extensible command grammar.

1. Introduction
---------------
Mathematical art creation requires specifying complex geometric
relationships, color mappings, and rendering parameters. A well-designed
DSL abstracts these concerns into composable primitives, allowing artists
and mathematicians to express visual ideas concisely. The polyart.lang
module implements such a language, providing both an interactive REPL
and a batch compilation mode.

2. DSL Design Principles
-------------------------
The polyart DSL follows these design principles:

  P1: Composability - Commands combine to build complex scenes.
  P2: Readability   - Syntax resembles mathematical notation.
  P3: Extensibility - New commands added without core changes.
  P4: Immediacy     - Changes visible without explicit compile step.
  P5: Precision      - Floating-point control with configurable epsilon.

The core vocabulary consists of 40 base commands, each mapping to a
function in the polyart.core module. Commands use prefix notation
for transformations and infix notation for arithmetic:

  curve y = 3*x^2 - 2*x + 1
  rotate(45deg) apply curve
  color gradient(blue, red) on curve

3. Tokenizer Architecture
--------------------------
The tokenizer converts raw source text into a stream of tokens.
Each token carries type, value, and position metadata.

Token types include:

  IDENTIFIER   : curve, rotate, scale, color, render, ...
  NUMBER       : 3.14159, -2.7, 1e-3
  OPERATOR     : +, -, *, /, ^, =, (, ), {, }, ;
  KEYWORD      : if, for, let, def, return, import
  STRING       : "filename.png"
  COMMENT      : // single line, /* multi-line */
  COMMA        : ,
  NEWLINE      : \n (significant for statement separation)

The tokenizer uses a deterministic finite automaton (DFA) implemented
in the polyart.lang.tokenizer module:

  class Tokenizer:
      def __init__(self, source):
          self.source = source
          self.pos = 0
          self.tokens = []

      def tokenize(self):
          while self.pos < len(self.source):
              self.scan_next_token()
          return self.tokens

      def scan_next_token(self):
          ch = self.source[self.pos]
          if ch.isdigit():
              self.scan_number()
          elif ch.isalpha() or ch == '_':
              self.scan_identifier()
          elif ch in '+-*/^=(){}[];,':
              self.scan_operator()
          elif ch == '"':
              self.scan_string()
          else:
              self.scan_whitespace()

The scanner handles multi-character tokens through maximum munch:
at each position, the longest matching token is selected. This
avoids ambiguity between operators (e.g., = vs == vs =>).

  4. Grammar Specification
-------------------------
The formal grammar uses Extended Backus-Naur Form (EBNF):

  program       = statement*
  statement     = assignment | command | block | if_stmt | for_stmt
  assignment    = IDENTIFIER '=' expression ';'
  command       = IDENTIFIER '(' arglist ')' ['on' expression]
  expression    = term (('+' | '-') term)*
  term          = factor (('*' | '/') factor)*
  factor        = base ('^' factor)?
  base          = NUMBER | IDENTIFIER | '(' expression ')'
                  | function_call | curve_literal
  function_call = IDENTIFIER '(' arglist ')'
  curve_literal = 'curve' '{' polynomial_spec '}'

5. Recursive Descent Parser
-----------------------------
The parser implements recursive descent, with each grammar
non-terminal mapping to a parse method. It produces an AST
using node classes from polyart.lang.ast:

  Parser.parse_program()   -> ProgramNode(statements)
  Parser.parse_statement() -> AssignmentNode | CommandNode | IfNode
  Parser.parse_expression()-> BinaryOpNode('+', left, right)
  Parser.parse_factor()    -> BinaryOpNode('^', base, exponent)

6. Abstract Syntax Tree
------------------------
The AST is a tree of typed nodes: ProgramNode, AssignmentNode,
CommandNode, BinaryOpNode, UnaryOpNode, NumberNode, IdentifierNode,
IfNode, ForNode, CurveNode, TransformNode, ColorNode. Each node
carries source position for error reporting.

7. Interpreter Architecture
-----------------------------
The interpreter traverses the AST, executing commands via a symbol
table and context stack. Commands are registered as function objects:

  @register('rotate')
  def cmd_rotate(angle_degrees, target=None):
      theta = math.radians(angle_degrees)
      return Transform('rotate', theta)

  @register('curve')
  def cmd_curve(coefficients, range_start=0, range_end=1):
      return PolynomialCurve(coefficients, (range_start, range_end))

8. Command Composition and Extensibility
-----------------------------------------
Commands compose through the 'on' keyword and transform chaining:

  rotate(30deg) on my_curve
  scale(2, 1) on rotate(30deg) on my_curve

The rightmost transform is applied first. Transform application
creates a new transformed object rather than modifying the original,
enabling non-destructive composition via the ComposeTransform class.

The DSL supports extension through several mechanisms:

  1. Import system: Load external command definitions.
  2. Function definitions: User-defined commands with def keyword.
  3. Plugin API: C-level extensions for performance-critical ops.
  4. Macro expansion: Compile-time code generation.

9. Error Handling and Performance
----------------------------------
The parser provides detailed error messages with source location,
using synchronization tokens to enable multiple error reporting
in a single pass. Key optimizations include constant folding,
bytecode compilation (15x speedup over tree-walking), cached
transform compositions, and lazy evaluation of curves.

10. Conclusion
--------------
A well-designed DSL for mathematical art significantly reduces the
cognitive overhead of specifying complex visual compositions. The
polyart.lang module demonstrates that recursive descent parsing,
tree-walking interpretation, and bytecode compilation can coexist
in a practical system. The extensibility mechanisms ensure the
language can grow with its community of users.

Future directions include visual programming interfaces, live
collaboration support, and integration with GPU shader compilation
for real-time rendering of mathematical art.
